# 01 任务结构规则

## 权威顺序

1. `sources/server-csharp-4.0.13/Libraries/SPTarkov.Server.Core/Models/Eft/Common/Tables/Quest.cs`
2. `sources/VisitAPI-main/Server/VisitApiQuestLoader.cs`
3. 官方 `quests.json` 真实样例

结论：VisitAPI 当前不会替你发明一套新的 quest 数据结构；它加载的是原生 SPT `Quest`。

## 关键规则

- quest 文件顶层通常是：
  - `{"questId": { ...QuestObject... }}`
- 外层 key、内部 `_id`、`templateId` 应保持一致。
- `traderId` 必须是合法商人 id。
- `type` 必须是合法任务类型，如 `Completion`。
- `side` 常见为 `Pmc`。
- `location` 常见可写 `any` 或具体地图 id。

## `HandoverItem` 规则

- `conditionType` 写 `HandoverItem`
- 上交目标写在 `target`
- `target` 可以是单个 TPL，也可以是 TPL 数组
- 数量写 `value`
- `onlyFoundInRaid` 明确写 `true/false`
- 不要写 `items`
- 不要写 `categories`

## 奖励规则

- 卢布奖励仍然走 `RewardType.Item`
- `items[0]._tpl` 写卢布 TPL：`5449016a4bdc2d6f028b456f`
- `upd.StackObjectsCount` 写金额
- `target` 指向奖励 items 中对应实例 `_id`

## locale 规则

每条 quest 至少准备这些 key：

- `{questId} name`
- `{questId} description`
- `{questId} note`
- `{questId} startedMessageText`
- `{questId} successMessageText`
- `{questId} failMessageText`
- `{questId} acceptPlayerMessage`
- `{questId} declinePlayerMessage`
- `{questId} completePlayerMessage`
- `{questId} changeQuestMessageText`

如果存在 `HandoverItem` 条件，还要补：

- `{conditionId}`

## 建议流程

1. 先用 `templates/quests/` 选模板。
2. 用 `indexes/item_tpl_index.jsonl` 或名称索引确认 TPL。
3. 用 `validators/validate_quest_json.py` 校验结构。
4. 用 `validators/validate_tpl_refs.py` 校验 TPL。
5. 用 `validators/validate_locale_keys.py` 校验本地化。
